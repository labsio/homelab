# FreshRSS — self-hosted RSS aggregator

Web-based feed reader with RSS-Bridge as a companion service for
generating feeds from sites that don't expose RSS natively.
Reached through Caddy at `rss.${DOMAIN}`, stores data in SQLite.

## Architecture

```
Caddy (rss.${DOMAIN}) ──► freshrss:80 ──► SQLite (freshrss_data volume)
                              │
                              └──► rss-bridge:80   (internal only)
```

- `freshrss` is on both `proxy_net` (for Caddy) and `rss_net` (for RSS-Bridge)
- `rss-bridge` is on `rss_net` only — not reachable from outside

## Prerequisites

- External Docker network `proxy_net` (shared with Caddy)
- Site block in `services/caddy/Caddyfile` routing `rss.${DOMAIN}` → `freshrss:80` (already present)
- DNS record `rss.${DOMAIN}` pointing at the Caddy host

## Setup

```bash
cp .env.example .env
docker compose up -d
```

Then open `https://rss.${DOMAIN}` and run the web installer:

- **Database**: SQLite (no credentials, no extra container)
- **Default user**: pick any username/password — this becomes the admin

## Using RSS-Bridge from FreshRSS

Inside FreshRSS, when adding a feed generated by RSS-Bridge, use the
internal hostname:

```
http://rss-bridge/?action=display&bridge=...&format=Atom
```

RSS-Bridge's own UI is intentionally not exposed through Caddy — browse
it locally if needed:

```bash
docker compose exec rss-bridge wget -qO- http://localhost/ | less
```

To expose RSS-Bridge publicly, add it to `proxy_net` and add a site
block to `Caddyfile` (e.g. `bridge.${DOMAIN}` → `rss-bridge:80`).

## Why SQLite

Single-user / small-homelab workloads do not benefit from a separate
DB container. SQLite keeps the stack to two services, one volume to
back up, and is what FreshRSS upstream recommends at this size.

## Why a separate `rss_net`

RSS-Bridge has no authentication. Keeping it off `proxy_net` means
only FreshRSS can reach it — other services on the shared proxy
network cannot call it, and it cannot be exposed by accident.

## Backup

The only state worth backing up is the `freshrss_data` volume — it
contains the SQLite DB, feed cache, and user settings. The
`freshrss_extensions` volume only matters if you have installed
third-party extensions.

```bash
docker run --rm \
  -v freshrss_freshrss_data:/data \
  -v "$PWD":/backup \
  alpine tar czf /backup/freshrss-$(date +%F).tar.gz -C /data .
```

## Notes

- `TRUSTED_PROXY` must include Caddy's IP range, otherwise FreshRSS
  ignores `X-Forwarded-*` and login / CSRF break behind the proxy.
- Feed refresh runs inside the container via `CRON_MIN` — no host
  cron needed.
- The official FreshRSS image ships its own HEALTHCHECK, no need
  to redefine one.
